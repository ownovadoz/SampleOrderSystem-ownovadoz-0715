# HARNESS.md — 서브에이전트 기반 개발 하네스

> **핵심 요약**: `docs/superpowers/plans/*.md`의 계획을 Task 단위로 서브에이전트에 위임해 개발한다. 각
> Task는 (1) 실패하는 기본 테스트만 커밋 → (2) 그 테스트를 통과시키는 구현 커밋, 이 두 단계로 진행한다.
> Debug/Release 빌드·테스트 실행 검증은 오케스트레이터/서브에이전트가 하지 않는다 — 계획 파일 하나가 끝나면
> 사용자에게 완료를 보고하고, 빌드 확인은 사용자가 직접 한다.

이 문서는 이 저장소에서 코드를 작성하는 모든 에이전트(오케스트레이터 + 서브에이전트)가 따라야 하는 개발
프로세스를 정의한다. 요구사항 자체(무엇을 만들지)는 `PRD.md`와 `docs/requirements/*.md`에 있고, 이 문서는
그것을 **어떻게** 만들지(프로세스)만 다룬다.

## 1. 서브에이전트 기반(subagent-driven) 개발

- 실제 구현 작업은 `docs/superpowers/plans/*.md`에 이미 Task 단위로 쪼개져 있다. 오케스트레이터는 계획을
  직접 구현하지 않고, Task 단위로 서브에이전트(Agent tool)에 위임한다.
- 커밋이 하나 생길 때마다(Red든 Green이든, 규칙 변경 커밋이든) 오케스트레이터는 곧바로 `git push`한다.
  여러 커밋을 모아뒀다가 한 번에 push하지 않는다.
- 한 Task를 위임할 때는 해당 Task의 `Files`/`Interfaces`/Step 목록 전체를 서브에이전트 프롬프트에 포함해
  자기완결적으로 만든다 — 서브에이전트는 계획 문서 전체나 이전 대화 맥락을 보지 못한다고 가정한다.
- 같은 모듈 내에서 여러 Task가 같은 파일(`*.vcxproj`, `*.vcxproj.filters`, `main.cpp` 등)을 순차적으로
  수정하므로, 한 모듈 안의 Task는 **순차적으로**(병렬 아님) 진행한다. 서로 다른 모듈(시료/주문/생산담당자/
  생산라인/모니터링)이 건드리는 파일이 겹치지 않는 구간에 한해서만 병렬 위임을 고려한다.
- 서브에이전트가 작업을 마치면, 오케스트레이터는 결과(빌드 로그, 테스트 출력, 실제 diff)를 직접 확인한다.
  다만 **사용자 확인은 Task마다 받지 않는다** — 같은 계획 파일 안의 Task를 연달아 서브에이전트에 위임해
  진행하고, 그 계획 파일의 모든 Task가 끝난 시점에 한 번만 사용자에게 결과를 요약해 보고하고 다음 계획으로
  넘어갈지 확인받는다. 단, 빌드/테스트가 실패해 서브에이전트가 스스로 해결하지 못하는 경우는 예외적으로
  즉시 사용자에게 알린다.

## 2. 빌드/테스트 검증은 사용자가 계획 단위로 직접 한다

- 오케스트레이터와 서브에이전트는 Debug/Release 빌드 성공 여부나 테스트 통과 여부를 **직접 실행해서
  확인하지 않는다.** `msbuild` 실행, 테스트 바이너리 실행, Release 콘솔 앱 수동 조작 등은 사용자의
  몫이다 — 특히 Release 실행 파일은 표준 입력을 기다리는 대화형 메뉴라 자동화된 확인이 실행을 멈추게
  만들 수 있으므로 더욱 그렇다.
- Task/계획 안에서는 코드 작성과 Red/Green 커밋까지만 진행한다. 같은 계획 파일 안의 모든 Task를
  순서대로 서브에이전트에 위임해 커밋까지 마치면, 그 계획은 "구현 완료" 상태다.
- 계획 파일 하나의 모든 Task 커밋이 끝나면 사용자에게 요약 보고하고, **사용자가 직접 빌드/테스트를
  확인한 뒤** 다음 계획으로 넘어갈지 알려준다. 오케스트레이터가 먼저 빌드해서 통과를 확인한 다음 보고하는
  것이 아니다.
- 단, 코드 작성 자체가 명백히 불가능한 상황(예: 계획에 없는 의존성이 필요함을 서브에이전트가 발견한 경우)은
  즉시 사용자에게 알린다.

## 3. TDD — Red 커밋과 Green 커밋을 분리

Task 안에서 테스트가 필요한 부분은 아래 2단계로 나누어 **각각 별도 커밋**을 남긴다.

1. **Red 커밋**: 아직 구현이 없어 컴파일이 실패하거나 assert가 실패하는, 그 기능의 핵심 동작만 검증하는
   "아주 잘 동작하는 기본 테스트"만 작성해 커밋한다. 모든 엣지 케이스를 한 번에 욱여넣지 않는다 — 우선
   해당 기능이 존재한다는 것과 정상 경로(happy path)만 검증하는 최소 테스트로 시작한다. **Red 상태에서
   실제로 컴파일이 실패하는지는 빌드해서 확인하지 않는다** — 빌드/테스트 확인 자체를 사용자가 계획 단위로
   직접 하기 때문이다 (2장 참고).
2. **Green 커밋**: 그 테스트를 통과시키는 최소 구현을 작성하고 커밋한다. 통과 여부도 오케스트레이터/
   서브에이전트가 확인하지 않고, 계획이 끝난 뒤 사용자가 직접 확인한다.
3. 이후 같은 Task 또는 다음 Task에서 엣지 케이스(경계값, 실패 경로, 동시성 등) 테스트를 추가할 때도 같은
   패턴(실패하는 테스트 커밋 → 통과시키는 구현 커밋)을 반복한다.
4. Red 커밋 없이 구현과 테스트를 한 커밋에 함께 넣지 않는다 — 테스트가 실제로 그 구현 없이는 실패한다는
   것을 이력으로 남기는 것이 목적이다.

## 4. 테스트 커버리지 미달 시 사용자에게 반드시 알릴 것

- 계획 문서(`docs/superpowers/plans/*.md`) 또는 모듈 요구사항 문서(`docs/requirements/*.md`)에 명시된
  항목 중 테스트로 검증하지 못한 항목이 있으면(시간 부족, 환경 제약, 외부 의존성 등 이유 불문) **작업을
  끝냈다고 보고하기 전에** 어떤 항목을 테스트하지 못했는지 사용자에게 명시적으로 알린다.
- "일부만 테스트했지만 잘 될 것 같다"는 식으로 넘어가지 않는다. 커버하지 못한 항목과 그 이유를 목록으로
  제시하고, 사용자가 그대로 진행할지 추가 작업을 할지 판단하게 한다.

## 5. Naming Convention

C++ 표준/커뮤니티에서 널리 쓰이는 방식(Google C++ Style Guide 계열)을 따르며, 기존
`docs/superpowers/plans/`에 이미 작성된 예시 코드(`Common/Clock.h` 등)의 스타일을 그대로 유지한다.

- 타입(클래스/구조체/enum class): `PascalCase` — 예: `Order`, `Sample`, `OrderStatus`, `IClock`,
  `SystemClock`, `FakeClock`
- 인터페이스(순수 가상 클래스): 접두사 `I` + `PascalCase` — 예: `IClock`
- 함수/메서드: `camelCase` — 예: `now()`, `advance()`, `set()`
- 지역 변수/파라미터: `camelCase` — 예: `initial`, `avgProductionTime`
- 멤버 변수(private): `camelCase` + 트레일링 언더스코어 — 예: `current_`
- 네임스페이스: `lower_case` — 예: `common`
- 파일명: `PascalCase.h` / `PascalCase.cpp` (타입 이름과 1:1 대응), 테스트 파일은 `<대상>Tests.cpp`
  — 예: `Clock.h`, `Clock.cpp`, `ClockTests.cpp`
- 매크로/상수 외에는 헝가리안 표기법을 쓰지 않는다.

## 6. PoC 저장소 활용

각 모듈을 구현하는 서브에이전트는 새로 설계를 고민하기 전에 먼저 아래 대응되는 PoC 저장소의 구조를 참고해
동등한 패턴으로 재구현한다 (import가 아니라 구조를 참고해 다시 작성하는 것). PoC 저장소는 이 저장소와
나란히 로컬에 클론돼 있다 (예: `C:\reviewer\DataPersistence-ownovadoz-0715`) — 실제 파일을 열어서 확인하지
않고 기억이나 추측으로 패턴을 재구현하지 않는다.

| 대상 | PoC 저장소 |
|---|---|
| MVC 계층 분리(Model/Controller/View) | `ConsoleMVC-ownovadoz-0715` |
| 데이터 영속성(저장/조회) | `DataPersistence-ownovadoz-0715` |
| 데이터 모니터링 도구 | `DataMonitor-ownovadoz-0715` |
| 더미 데이터 생성(테스트 전용) | `DummyDataGenerator-ownovadoz-0715` |

`PRD.md` 3장(아키텍처 공통 규칙)과 상충하지 않는지 항상 함께 확인한다.

### 6.1 계획서를 그대로 믿지 않는다 (재발 방지)

`docs/superpowers/plans/*.md`는 PoC 4개가 준비되기 전이나 그 내용을 충분히 반영하지 못한 상태로 작성된
경우가 있었다 — 실제로 sample-clerk/order-clerk 계획서가 `DataPersistence` PoC(JSON) 대신 파이프 구분
텍스트 포맷을 자체적으로 설계해서 PRD 3장을 위반한 사례가 있었다. 이런 일이 다시 생기지 않도록:

- 오케스트레이터는 계획서의 한 Task를 서브에이전트에 위임하기 **전에**, 그 Task가 건드리는 관심사가 위 표의
  어느 PoC에 대응하는지 먼저 판단하고, 대응되는 PoC가 있으면 그 PoC의 실제 파일을 최소 한 번은 읽어 계획서
  코드가 그 구조와 맞는지 확인한다. 안 맞으면(계획서가 자체 설계를 담고 있으면) 그대로 위임하지 말고, 먼저
  사용자에게 알리고 방향을 정한 뒤 진행한다.
- 특히 파일 저장/조회(영속성)는 항상 `DataPersistence`의 `JsonValue`/`FileRepository<T>` 패턴을 써야 한다
  (파이프 구분 텍스트, CSV 등 자체 포맷 금지).
- 모니터링 모듈은 `DataMonitor` PoC 구조를 반드시 따른다 (검색 기능 포함, PRD 3장 참고).
- **gtest 단위 테스트의 전제조건(precondition) 데이터도 `Testing/DummyDataGenerator`(`DummyDataGenerator`
  PoC를 결정적 시드로 재구현한 것, `Testing/DummyDataGenerator.h`)를 통해 만든다.** "그 시료/주문이 어떤
  내용이든 상관없이 그냥 하나 있으면 되는" 전제조건 데이터(예: 주문을 만들기 위해 등록해두는 시료)는 전부
  `DummyDataGenerator gen; auto s = gen.sample(1);` 형태로 생성한다. 반대로 **그 값 자체가 테스트의 검증
  대상인 리터럴은 그대로 손으로 쓴다** — 예: 유효하지 않은 수율 경계값(1.5, 0.0), 검색 대소문자 무시 매칭에
  쓰이는 특정 문자열, View 테스트에서 사용자가 입력한 것처럼 시뮬레이션하는 텍스트, 재고/예약수량 합산 같은
  산술 검증에 쓰이는 구체적인 수량 값. 새로 테스트를 추가할 때도 이 구분 기준으로 판단한다.

### 6.2 메인 메뉴 구조: 모듈별 대분류 → 세부 메뉴 (2단계)

메인 메뉴는 모든 항목을 한 화면에 나열하는 1단계 평면 메뉴가 아니라, **시료 관리 / 주문(접수/승인/거절) /
모니터링 / 출고 처리 / 생산 라인** 5개 대분류를 보여주고, 하나를 선택하면 그 모듈의 세부 메뉴(등록/조회/
검색, 예약/승인/거절 등)로 들어가는 2단계 구조를 따른다 (`main.cpp`의 `runSampleMenu`/`runOrderMenu`처럼
모듈별 서브메뉴 함수를 두고, 메인 루프는 그 함수를 호출만 한다). 아직 구현되지 않은 대분류를 선택하면
"아직 구현되지 않았습니다" 안내만 보여주고 메인 메뉴로 돌아간다. 계획서(`docs/superpowers/plans/*.md`)에
남아 있는 "메인 메뉴에 항목을 이어서 추가한다"는 옛 1단계 평면 메뉴 서술은 이 2단계 구조로 대체된 것으로
간주하고, 각 모듈 구현 시점에 main.cpp를 이 구조에 맞게 반영한다.

## 7. 이 문서와 다른 문서의 관계

- **무엇을 만들지**(도메인 규칙, 모듈별 요구사항): `PRD.md`, `docs/requirements/*.md`
- **어떤 순서로 구현할지**(Task 목록, 파일별 Step): `docs/superpowers/plans/*.md`
- **어떻게 개발할지**(이 문서, 프로세스/컨벤션): `HARNESS.md`

세 문서가 상충하는 것처럼 보이면, 요구사항 원본(`docs/requirements.pdf`, `docs/memo.txt`)을 기준으로
판단하고 그 결정을 계획 문서에 반영한다.
