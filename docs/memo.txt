- 재고에 추가되는 건 주문에 대한 생산이 완료됐을 때 (총 생산시간이 끝나면)
- 시간은 현실 시간 반영 (테스트에는 시간을 임의로 변경할 수 있어야 함)
- 중간에 프로그램 종료 시 재시작하면 진행상황이 그대로 반영되어야 함
	-- 종료 시 전체 내용 저장, 실행 시 복원 및 반영 완료
	-- 예를 들어 생산 중이던게 완료되는 시점 넘어 프로그램이 켜졌다면 생산 완료되어야 함
- 생산라인은 수율을 나눈 양으로 총 생산한다.
- 주문시에는 생산 중인 건 고려하지 않고, 현재 선점되지 않은 재고를 기준으로만 생산 요청한다.
	-- 주문담당자 - 생산담당자 - 생산라인에서 주문담당자와 생산라인의 직접적인 연결은 없다는 뜻
- 생산은 큐에 들어오면 무조건 생산, 처리 순서는 FIFO로 한다.
	-- 얘는 큐에 있는지, 있는 건 생산하고 완료되면 재고로 업데이트 함
	-- 근데 직접하는 게 아니고 요청이 왔을 때만 하면 될듯
	-- 멀티 스레딩 이런 거 안쓸꺼니까
- 생산라인은 한 번에 제품 1개씩만 생성 가능하다. (이것도 저장 및 복원되어야 한다.)
- 재고를 볼 때는 다른 주문에서 선점한 걸 고려해서 봐야 함
- 생산 시작에는 수율을 반영해서 더 많이 생산하지만, 생산 완료 시점에서는 수율 100%로 생성된다. (그래서 재고가 쌓이는 구조)
- 출고는 사람이 직접하므로 생산이 완료됐다고 released로 변경하면 안된다.
	-- 어쨌든 재고는 이미 선점되어있는 상태이기 때문에 재고 계산 등에는 문제 없음
- 다만, 선점돼있는 재고더라도 재고 현황에서는 재고로 보여야 한다.
	-- 재고 파악때는 재고 그대로, 다만 주문이 들어왔을 때는 선점된 건 제외하고 봐야 함
- 주문 받았을 때, 재고를 확인할 때, 생산 라인의 현 상황이 update 되어야 할 것
	-- 혹은 더 좋은 때가 있는지? 서로 언제 정보를 업데이트 할 지?

- 개발에 필요한 것들은 스킬로 관리하고, 요구사항에 대한 건 멀티 에이전트로 구분
- PoC를 그대로 쓰는 건 아니지만 그 기반을 사용해야 함. (라이브러리화는 아니되, PoC를 바탕으로 코드 만들어 달라고 해야 함)
