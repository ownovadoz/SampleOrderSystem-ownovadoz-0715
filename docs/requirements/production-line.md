# 생산 라인 모듈 요구사항

이 문서는 생산 라인 모듈 개발을 담당하는 에이전트를 위한 문서입니다. 시스템 공통 배경, 도메인 모델(시료/
주문 필드, 주문 상태, 재고 선점 규칙), 시간/영속성 정책, 아키텍처·개발 공통 규칙은 `PRD.md`를 먼저 읽고
따릅니다. 이 문서에는 생산 라인 모듈에만 해당하는 요구사항만 적습니다.

각 항목 끝의 "출처"는 사람이 원문을 다시 확인해 검증할 수 있도록, 어떤 문서의 어느 부분을 근거로 정리했는지
표시한 것입니다.

## 목적

> **핵심 요약**: 승인 시점에 부족했던 물량을 실제로 생산해 재고를 채우는 단일 생산 설비.

생산 라인은 공장에서 시료 하나를 생산하는 설비 흐름이다. 주문 승인 시점에 재고가 부족했던 분량을 실제로
생산하여 재고를 채운다.

> 출처: `docs/requirements.pdf` p.7 (시스템 개요 — "생산 라인: 공장에서 시료 하나를 생산하는 설비 흐름")

## 전제 조건 (설비 특성)

> **핵심 요약**: 생산 라인 1개, 동시 생산 1개, FIFO, 멀티스레딩 없음(조회 시점에 지연 계산).

- 생산 라인은 **1개**이며, 한 번에 시료를 **1개씩만** 생산할 수 있다.
- 주문이 들어온(승인되어 생산이 필요해진) 시료에 대해서만 생산한다.
- 생산 큐에 들어온 작업은 조건 없이 무조건 생산하며, 처리 순서는 **FIFO**다.
- 멀티스레딩을 사용하지 않는다. 생산 진행/완료는 주문 승인, 재고 조회, 생산 라인 조회 등 관련 동작이
  호출되는 시점에 경과 시간을 계산해 지연 반영(lazy evaluation)한다 (`PRD.md` 2.5절 참고).

> 출처: `docs/requirements.pdf` p.7 (시스템 개요 — "하나의 생산 라인은 시료를 하나씩 생산", "주문이 들어온
> 시료에 대해서만 생산"), p.20 (생산라인 — "스케쥴링 전략 : FIFO") / `docs/memo.txt` 13번째 줄 ("생산라인은
> 한 번에 제품 1개씩만 생성 가능하다"), 9번째 줄 ("생산은 큐에 들어오면 무조건 생산, 처리 순서는 FIFO로
> 한다"), 11~12번째 줄 ("근데 직접하는 게 아니고 요청이 왔을 때만 하면 될듯", "멀티 스레딩 이런 거
> 안쓸꺼니까")

> 확인"), p.20 (메뉴별 상세)

## 상세 규칙

### 생산 요청 접수 (주문 관리 모듈로부터)

> **핵심 요약**: 주문 관리 모듈이 넘긴 (시료ID, 부족분, 주문ID)를 FIFO 큐 맨 뒤에 추가.

주문 관리 모듈이 주문 승인 시 재고 부족을 판단하면, 아래 정보를 담아 생산 요청을 전달한다:
- 시료 ID
- 부족분 수량 (주문 수량 − 승인 시점의 가용 재고)
- 연관된 주문 ID

이 요청을 생산 큐의 맨 뒤에 추가한다 (FIFO).

> 출처: `docs/requirements.pdf` p.16 (주문 승인/거절 — "재고가 부족한 경우 → 생산 라인에 자동으로 등록"),
> p.21 (예시 UI — 대기 중인 주문 표: 주문번호/시료/주문량/부족분)

### 실 생산량/생산 시간 계산

> **핵심 요약**: 실 생산량 = ceil(부족분 / 수율), 총 생산 시간 = 평균 생산시간 × 실 생산량.

- 실 생산량 = `ceil(부족분 / 수율)` — 수율을 반영해 부족분보다 많이 생산한다.
- 총 생산 시간 = `평균 생산시간 * 실 생산량`
- 수율/평균 생산시간은 해당 시료의 등록된 속성값을 사용한다 (시료 관리 모듈에서 조회).

> 출처: `docs/requirements.pdf` p.20 (생산라인 — "실 생산량 : ceil(부족분 / 수율)", "총 생산 시간 : 평균
> 생산시간 * 실 생산량"), p.21 (예시 UI — "실생산량 206 ea / 165 min" 계산 예시) / `docs/memo.txt` 6번째 줄
> ("생산라인은 수율을 나눈 양으로 총 생산한다")

### 생산 완료 처리

> **핵심 요약**: 완료 시 실 생산량만큼 재고 증가(수율 100%로 인정) + 연관 주문 PRODUCING→CONFIRMED.

생산 시작 시각 + 총 생산 시간이 지나면 해당 작업은 완료된 것으로 간주한다 (조회 시점에 경과 시간을 계산해
판정, 백그라운드 타이머 없음). 완료 시:

1. 해당 시료의 재고를 **실 생산량만큼** 증가시킨다. 수율은 이미 생산 시작 시 반영했으므로, 완료 시점에는
   실 생산량 전량이 정상품(수율 100%)으로 재고에 더해진다 (이 때문에 재고가 시간이 갈수록 쌓이는 구조가
   된다).
2. 연관된 주문의 상태를 `PRODUCING` → `CONFIRMED`로 전환하기 위해 생산담당자에게 생산 완료를 알린다.
    생산 라인은 생산만 담당하지 주문 상태를 직접 바꾸는 역할을 하진 않는다.
3. 큐의 다음 작업(있다면)의 생산을 이어서 시작한다.

> 출처: `docs/requirements.pdf` p.20 (생산라인 — "생산 완료시 주문상태 PRODUCING -> CONFIRMED 변경") /
> `docs/memo.txt` 1번째 줄 ("재고에 추가되는 건 주문에 대한 생산이 완료됐을 때 (총 생산시간이 끝나면)"),
> 15번째 줄 ("생산 시작에는 수율을 반영해서 더 많이 생산하지만, 생산 완료 시점에서는 수율 100%로 생성된다.
> (그래서 재고가 쌓이는 구조)"), 10번째 줄 ("있는 건 생산하고 완료되면 재고로 업데이트 함")

### 여러 작업이 한 번에 완료 판정되는 경우 (catch-up)

> **핵심 요약**: 프로그램이 꺼져 있던 사이 여러 작업이 완료 시각을 지났다면, FIFO 순서대로 순차 완료
> 처리해 모두 최신 상태로 만든다.

프로그램이 오래 꺼져 있었거나 조회가 뜸했던 경우, 한 번의 조회 시점에 큐에 있던 작업 여러 개가 이미
완료 시각을 지났을 수 있다. 이 경우 큐 순서(FIFO)대로 하나씩 순차적으로 완료 처리하며 (즉, 완료된 작업의
소요 시간만큼 다음 작업의 시작 시각을 앞당겨 누적 계산), 그 다음 아직 진행 중이거나 대기 중인 작업까지
모두 최신 상태로 반영한다.

> 출처: `docs/memo.txt` 3~5번째 줄 ("중간에 프로그램 종료 시 재시작하면 진행상황이 그대로 반영되어야 함",
> "예를 들어 생산 중이던게 완료되는 시점 넘어 프로그램이 켜졌다면 생산 완료되어야 함")

### 생산 현황 표기

> **핵심 요약**: 표시 정보 수준은 자유. 예: 주문 정보, 진행률.

현재 생산 중인 작업에 대해 표시할 정보 수준은 자율적으로 결정한다 (예: 연관 주문 정보, 시작 시각, 현재까지
경과/진행률, 예상 완료 시각 등).

> 출처: `docs/requirements.pdf` p.20 (생산라인 — "표기할 정보 수준은 자율적으로 결정, ex) 주문 정보,
> 현재까지의 생산량 등"), p.21 (예시 UI — 진행률/완료 예정 시각 표시)

### 대기 주문 확인

> **핵심 요약**: FIFO 순서로 대기 큐 표시.

생산 큐에 대기 중인 작업 목록을 FIFO 순서로 표시한다.

> 출처: `docs/requirements.pdf` p.20 (생산라인 — "대기 주문 확인: 생산라인의 대기열인 생산 큐를 이용, 생산
> 작업을 대기하고 있는 목록을 출력, 스케쥴링 전략 : FIFO")

## 다른 모듈과의 인터페이스 계약

- 주문 관리 모듈로부터 "부족분 생산 요청(시료 ID, 부족분 수량, 주문 ID)"을 받는 인터페이스를 제공한다.
- 생산 완료 시 주문 관리 모듈이 정의한 주문 상태(`PRODUCING` → `CONFIRMED`)를 갱신할 수 있어야 한다
  (공유 도메인 모델의 주문 엔티티를 직접 갱신하거나, 주문 모듈이 제공하는 상태 갱신 인터페이스를 사용).
- 시료 관리 모듈로부터 시료의 평균 생산시간/수율 속성을 조회하고, 생산 완료 시 재고를 증가시킨다 (재고
  증가는 시료 관리 모듈이 관리하는 재고 데이터에 반영되어야 한다).
- 모니터링 모듈이 생산 큐/현재 진행 상황을 다시 보여줄 수도 있으므로, 생산 라인 상태 조회 기능은 다른
  모듈에서도 재사용 가능해야 한다.
