# 주문 접수/승인/거절 모듈 요구사항

이 문서는 주문(접수/승인/거절) 모듈 개발을 담당하는 에이전트를 위한 문서입니다. 시스템 공통 배경, 도메인
모델(시료/주문 필드, 주문 상태, 재고 선점 규칙), 시간/영속성 정책, 아키텍처·개발 공통 규칙은 `PRD.md`를
먼저 읽고 따릅니다. 이 문서에는 주문 모듈에만 해당하는 요구사항만 적습니다.

각 항목 끝의 "출처"는 사람이 원문을 다시 확인해 검증할 수 있도록, 어떤 문서의 어느 부분을 근거로 정리했는지
표시한 것입니다.

## 목적

> **핵심 요약**: 고객 요청 → 주문 접수(RESERVED) → 생산 담당자 승인/거절.

고객이 시료를 요청하면 주문 담당자가 주문을 생성(접수)하고, 생산 담당자가 그 주문을 승인하거나 거절한다.

> 출처: `docs/requirements.pdf` p.14 (시료 주문 — "고객이 시료를 요청하면 주문 담당자가 주문을 생성"), p.6
> (역할별 흐름도)

## 메뉴 구성

| 메뉴 | 설명 |
|---|---|
| 시료 예약(주문 접수) | 고객이 원하는 시료와 수량으로 주문 생성 |
| 접수된 주문 목록 | RESERVED 상태의 주문 목록 표시 |
| 주문 승인 | 접수된 특정 주문을 승인 |
| 주문 거절 | 접수된 특정 주문을 거절 |

> 출처: `docs/requirements.pdf` p.10 (메인 메뉴 — "주문 (접수/승인/거절): 고객 주문 접수 및 생산 라인
> 담당자의 승인·거절 처리"), p.14, p.16 (메뉴별 상세)

## 상세 규칙

### 시료 예약(주문 접수)

> **핵심 요약**: 시료ID·고객명·주문수량을 입력받아 RESERVED 상태로 생성. 이 시점에는 재고를 확인하지 않는다.

입력 값:
- 시료 ID (시료 관리 모듈에 등록되어 있어야 하며, 존재하지 않는 시료 ID는 접수를 거부한다)
- 고객명
- 주문 수량 (1 이상)

주문 생성 시 상태는 `RESERVED`로 설정한다. 이 시점에는 재고 확인/선점을 하지 않는다 — 재고 확인은 승인
시점에만 이루어진다 (`PRD.md` 2.4절 참고).

> 출처: `docs/requirements.pdf` p.14 (시료 주문 — "예약시 입력 값: 시료 ID, 고객명, 주문 수량", "이 시점에서
> 주문 상태는 RESERVED") / `docs/memo.txt` 7번째 줄 ("주문시에는 생산 중인 건 고려하지 않고, 현재 선점되지
> 않은 재고를 기준으로만 생산 요청한다" — 재고 확인은 접수가 아닌 승인 시점의 일임을 뒷받침)

### 접수된 주문 목록

> **핵심 요약**: RESERVED 상태 주문만 표시.

`RESERVED` 상태의 주문만 목록으로 표시한다.

> 출처: `docs/requirements.pdf` p.16 (주문 승인/거절 — "접수된 주문 목록: RESERVED 상태의 주문 목록 Display")

### 주문 승인

> **핵심 요약**: 가용 재고가 충분하면 즉시 CONFIRMED, 부족하면 부족분만 생산 라인에 요청하고 PRODUCING으로
> 전환.

특정 `RESERVED` 주문을 승인 처리한다. 처리 절차:

1. 해당 시료의 **가용 재고**를 계산한다: `가용 재고 = 원재고 - 해당 시료의 선점된 재고 합`
   (선점된 재고 = `CONFIRMED`/`PRODUCING` 상태인 다른 주문들의 수량 합. 자세한 정의는 `PRD.md` 2.4절 참고)
2. 가용 재고가 주문 수량 이상이면 → 주문 상태를 즉시 `CONFIRMED`로 전환한다.
3. 가용 재고가 주문 수량보다 부족하면:
   - 부족분 = 주문 수량 − 가용 재고
   - 부족분을 생산 라인의 생산 큐에 등록 요청한다 (시료 ID, 부족분 수량, 이 주문 ID를 전달).
     생산 라인 모듈이 실제 생산량/생산 시간 계산과 큐 처리를 담당하므로, 이 모듈은 "부족분 생산을 요청"하는
     역할만 하고 생산 알고리즘 자체에 관여하지 않는다 (경계는 `PRD.md` 1장 참고).
   - 주문 상태를 `PRODUCING`으로 전환한다.
   - 이후 생산 라인에서 해당 생산 작업이 완료되면, 생산 라인 모듈이 이 주문을 `PRODUCING` → `CONFIRMED`로
     전환한다 (생산 라인이 생산 담당자에게 notify만 보내고 실제 처리는 생산 담당자가 해야 한다.).

> 출처: `docs/requirements.pdf` p.16 (주문 승인/거절 — "승인시 재고 상황에 따라 2가지 방식으로 자동으로
> 처리: 재고가 충분한 경우 → 즉시 CONFIRMED / 재고가 부족한 경우 → 생산 라인에 자동으로 등록, PRODUCING으로
> 전환"), p.17 (예시 UI — 부족분 계산 예시), p.20 (생산라인 — "실 생산량 : ceil(부족분 / 수율)") /
> `docs/memo.txt` 7번째 줄 (승인 판단은 미선점 재고 기준)

### 주문 거절

> **핵심 요약**: RESERVED → REJECTED. 이후 모든 집계에서 제외.

특정 `RESERVED` 주문을 즉시 `REJECTED` 상태로 전환한다. 이후 이 주문은 모든 목록/집계에서 제외된다.

> 출처: `docs/requirements.pdf` p.16 (주문 승인/거절 — "주문 거절: 접수된 특정 주문에 대해 거절, 즉시
> REJECTED 상태로 전환"), p.8 ("REJECTED는 거절된 주문으로 정상 흐름 외의 상태이며 모니터링에서 제외")

`docs/requirements.pdf` p.15 (시료 주문), p.17 (주문 승인/거절)을 참고한다. 다만 화면 문구/표 형식 등

## 다른 모듈과의 인터페이스 계약

- 시료 관리 모듈로부터 시료 존재 여부/속성 조회 기능을 사용한다.
- 생산 라인 모듈에 "부족분 생산 요청(시료 ID, 부족분 수량, 주문 ID)"을 전달할 수 있는 인터페이스가
  필요하다. 생산 라인 모듈은 이 요청을 FIFO 큐에 등록하고, 완료 시 해당 주문을 `CONFIRMED`로 전환한다.
- 모니터링/출고 모듈은 주문 목록과 상태를 조회하므로, 주문 데이터(주문번호/시료ID/고객명/수량/상태) 조회
  기능이 다른 모듈에서도 재사용 가능해야 한다.
