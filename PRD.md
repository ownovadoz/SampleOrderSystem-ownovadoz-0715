# PRD.md

반도체 시료 생산주문관리 시스템의 **모든 모듈(에이전트)이 공통으로 지켜야 하는** 요구사항만 정리한 문서입니다.
특정 모듈 1~2개에만 관련된 내용은 여기 두지 않고 각 모듈의 요구사항 문서(`docs/requirements/*.md`)에 각자 기술합니다.
(두 모듈에 걸쳐 겹치는 내용이라도 이 문서로 옮기지 않고, 각 모듈 문서에 중복 기술하는 것을 원칙으로 합니다.)

각 항목 끝의 "출처"는 사람이 원문을 다시 확인해 검증할 수 있도록, 어떤 문서의 어느 부분을 근거로 정리했는지
표시한 것입니다.

## 1. 역할 및 처리 흐름

> **핵심 요약**: 주문 담당자 → 생산 담당자 순으로 흐르며, 주문 담당자와 생산 라인은 서로 직접
> 연결되어 있지 않다.

- **주문 담당자**: 요청에 맞게 주문서를 작성 (시료 예약/접수 처리)
- **생산 담당자**: 주문 수신 후 승인 또는 거절 처리, 생산 라인을 운용

주문 담당자와 생산 라인 사이에는 **직접적인 연결이 없다.** 즉, 주문을 접수/승인하는 로직은 재고 수량만을
근거로 판단하며, 생산 라인의 현재 진행 상황(무엇을 생산 중인지)을 직접 참조하지 않는다. 반대로 생산 라인은
승인 시점에 큐에 등록된 작업만 알며, 주문 접수 자체에는 관여하지 않는다. 이 경계는 각 모듈을 독립된
에이전트가 개발하기 위한 전제이기도 하므로 반드시 지켜야 한다.

> 출처: `docs/requirements.pdf` p.6 (역할별 흐름도) / `docs/memo.txt` 8번째 줄 ("주문담당자 - 생산담당자 -
> 생산라인에서 주문담당자와 생산라인의 직접적인 연결은 없다는 뜻")

## 2. 공통 도메인 모델 (Single Source of Truth)

모든 모듈이 동일한 데이터 구조와 상태 정의를 공유해야 하므로, 아래 모델은 어느 모듈이 구현하든 임의로
변경하지 않는다.

### 2.1 시료 (Sample)

> **핵심 요약**: 시료 ID·이름·평균 생산시간·수율 4개 속성. 수율은 (정상품 수 / 총 생산 수).

| 필드 | 설명 |
|---|---|
| 시료 ID | 고유 식별자 |
| 이름 | 시료명 |
| 평균 생산시간 | 1개 생산에 걸리는 평균 시간 |
| 수율 | (정상적인 시료 수 / 총 생산 시료 수), 0~1 사이 값. ex) 100개 생산 중 90개 정상 = 0.9 |

### 2.3 주문 상태 및 전이

> **핵심 요약**: RESERVED(접수) → CONFIRMED(출고대기) 또는 PRODUCING(생산중) → CONFIRMED → RELEASE(출고완료).
> REJECTED는 모니터링에서 항상 제외. RELEASE 전환은 반드시 사람이 직접 실행해야 한다.

| 상태 | 의미 |
|---|---|
| RESERVED | 주문 접수 |
| REJECTED | 주문 거절 (정상 흐름 밖의 최종 상태, **모든 모니터링/집계에서 항상 제외**) |
| PRODUCING | 주문 승인 완료, 재고 부족으로 생산 중 |
| CONFIRMED | 주문 승인 완료, 출고 대기 중 |
| RELEASE | 출고 완료 |

```
RESERVED --(거절)--> REJECTED                          [최종 상태, 모니터링 제외]
RESERVED --(승인, 재고 충분)--> CONFIRMED
RESERVED --(승인, 재고 부족)--> PRODUCING --(생산 완료)--> CONFIRMED
CONFIRMED --(사람이 직접 출고 처리)--> RELEASE
```

- `RELEASE`로의 전환은 **항상 사람이 직접 트리거하는 동작**이다. 생산 완료, 재고 충족 등 어떤 자동 로직도
  주문을 `RELEASE`로 바꿀 수 없다.
- 상태 값의 철자/명칭을 모듈마다 다르게 쓰지 않는다 (예: `RELEASE`를 `RELEASED`로 바꾸는 등 임의 변형 금지).

> 출처: `docs/requirements.pdf` p.8 (주문 상태 흐름 표), p.7 (상태 전이 다이어그램), p.16 (주문 승인/거절 —
> 재고 충분/부족에 따른 자동 분기) / `docs/memo.txt` 16번째 줄 ("출고는 사람이 직접하므로 생산이 완료됐다고
> released로 변경하면 안된다")

### 2.4 재고 선점(reservation) 규칙

> **핵심 요약**: 재고는 생산 완료 때 늘고 출고 때 준다. "선점된 재고"는 CONFIRMED/PRODUCING 주문이 장차
> 쓸 물량이며, 신규 주문 승인 판단에는 이를 뺀 가용 재고를 쓰지만, 재고 조회 화면에는 항상 원재고 그대로
> 보여준다.

- 재고의 물리적 수량은 **생산 완료 시 증가**하고 **출고(RELEASE) 처리 시 감소**한다. 그 외 시점에는 변하지
  않는다.
- "선점된 재고"란 승인 완료 상태(`CONFIRMED` 또는 `PRODUCING`)이면서 아직 출고되지 않은 주문들이 장차
  소비할 예정인 수량이다 (시료별 해당 주문 수량의 합).
- 주문은 수율을 고려해서 더 많이 생산하고, 실제 생산 시 수율은 100%이기 때문에 물량이 더 남을 수 있다.
  이때는 실제 주문한 물량만 선점으로 표시되고 나머지는 선점되지 않은 재고로 남아 있어야 한다.
- **신규 주문을 승인할지 판단할 때**는 항상 다음 값을 기준으로 한다:
  `가용 재고 = 원재고 - 해당 시료의 선점된 재고 합`
  아직 승인되지 않은 `RESERVED` 주문이나 생산 라인의 실시간 진행 상황은 이 계산에 포함하지 않는다.
- **재고 현황을 조회/표시할 때**는 선점 여부와 무관하게 항상 **원재고(물리적 수량)** 를 그대로 보여준다.
  선점 여부는 승인 판단에만 영향을 주고, "재고가 얼마나 있는지" 화면에는 영향을 주지 않는다.
- 재고를 확인할 시점에는 먼저 생산 라인에서 이 시간에 샘플이 다 생산되었는지 먼저 확인하고 재고 수를 업데이트 해야한다.

> 출처: `docs/memo.txt` 7번째 줄 ("주문시에는 생산 중인 건 고려하지 않고, 현재 선점되지 않은 재고를
> 기준으로만 생산 요청한다"), 14번째 줄 ("재고를 볼 때는 다른 주문에서 선점한 걸 고려해서 봐야 함"),

### 2.5 시간 및 영속성 공통 정책

> **핵심 요약**: 현실 시간을 쓰되 테스트에서는 모킹 가능해야 한다. 종료 시 전체 상태(진행 중인 생산 포함)를
> 저장하고, 재시작 시 경과 시간만큼 자동으로 따라잡는다(catch-up). 백그라운드 스레드/타이머는 쓰지 않는다.

- 시간은 현실 시간을 그대로 반영하되, **테스트 시 임의로 변경/모킹할 수 있어야 한다.** 따라서 어떤 모듈도
  `std::chrono::system_clock::now()` 등을 직접 호출하지 않고, 공용 시간 소스(Clock 인터페이스 등)를 통해서만
  현재 시각을 얻는다.
- 애플리케이션 종료 시 전체 도메인 상태(시료, 재고, 주문, 생산 큐 및 **생산 라인에서 진행 중인 작업의
  진행 상황 포함**)를 저장하고, 재시작 시 그대로 복원한다.
- 각 도메인 정보들은 별도 파일로 저장하고, 다른 모듈은 접근할 수 없도록 한다.
- 재시작 시, 종료되어 있던 동안 실제 시간이 얼마나 흘렀는지를 계산하여 그 사이 완료됐어야 할 생산 작업은
  즉시 완료 처리(catch-up)한다. 예: 생산 중이던 작업이 끝나는 시점을 지나 프로그램이 다시 켜졌다면, 시작하자마자
  생산 완료로 반영되어야 한다.
- 상태 갱신(생산 진행/완료 등)은 백그라운드 스레드나 타이머로 하지 않는다. 관련 동작이 호출되는 시점
  (주문 승인, 재고 조회, 생산 라인 조회 등)에 그때까지 경과한 시간을 계산해 지연 반영(lazy evaluation)한다.
  멀티스레딩을 사용하지 않는다.

> 출처: `docs/memo.txt` 2번째 줄 ("시간은 현실 시간 반영 (테스트에는 시간을 임의로 변경할 수 있어야 함)"),
> 3~5번째 줄 ("중간에 프로그램 종료 시 재시작하면 진행상황이 그대로 반영되어야 함" 및 catch-up 예시),
> 12번째 줄 ("멀티 스레딩 이런 거 안쓸꺼니까"), 13번째 줄 ("생산라인은 한 번에 제품 1개씩만 생성 가능하다.

## 3. 아키텍처 공통 규칙

> **핵심 요약**: MVC 구조, PoC 저장소 4개(ConsoleMVC/DataPersistence/DataMonitor/DummyDataGenerator)의
> 구조를 참고해 다시 구현. 모니터링 도구는 검색 기능 필수.

- Model / Controller / View 계층을 분리하는 MVC 구조를 따르며, 구조 자체는 `ConsoleMVC-ownovadoz-0715` PoC를
  기준으로 확장한다.
- 데이터 저장/조회(영속성) 구현 방식은 `DataPersistence-ownovadoz-0715` PoC의 방식을 그대로 따른다.
- 콘솔에서 저장된 데이터 상태를 실시간 조회하는 모니터링 도구는 `DataMonitor-ownovadoz-0715` PoC의 구조를
  기준으로 하되, 검색 기능을 포함해야 한다.
- 테스트용 더미 데이터 생성 방식은 `DummyDataGenerator-ownovadoz-0715` PoC의 구조를 참고한다.
- 시스템은 콘솔 기반으로 동작하며, 담당자가 메뉴에서 번호 등을 입력해 기능을 실행하는 방식이다. 화면 UI의
  구체적인 표현(문구, 표 형식 등)은 자유롭게 결정한다.
- 테스트용 더미 데이터 생성은 테스트 돌릴 때만 사용하고, 랜덤을 사용하는 경우 시드 값을 이용해 항상 테스트 결과가 일정하도록 해야 한다.
- 실제 실행 시에는 항상 담당자가 직접 데이터를 입력해야 하기에 테스트용 더미 데이터 생성이 사용되어선 안된다.
- 각 모듈(에이전트)은 공통 도메인 모델(3장)을 그대로 재사용하고, 자신이 담당하는 메뉴의 Controller/View만
  구현한다. 도메인 모델(엔티티 정의, 상태 enum 등) 자체를 모듈별로 임의 확장/변형하지 않는다.
